Overview:

ITERATION 1 — RANDOM FOREST (Baseline)
----------------------------------------------------------------

What is Random Forest?
Random Forest builds hundreds of independent decision trees and
averages their predictions. Each tree looks at a random subset
of the data and features, then votes on the final answer.
Think of it as asking 100 different experts and taking the
majority opinion.

Why we chose it first:
- Well known and widely used algorithm
- Handles mixed feature types well (numerical, binary, encoded)
- Works great with large datasets
- Gives feature importance out of the box
- Easy to justify and explain

Settings used:
- n_estimators = 100 (100 decision trees)
- random_state = 42 (for reproducibility)
- n_jobs = -1 (use all CPU cores)

ITERATION 2 — XGBOOST (Upgrade from Random Forest)
----------------------------------------------------------------

What is XGBoost?
XGBoost stands for Extreme Gradient Boosting. Unlike Random
Forest which builds trees independently, XGBoost builds trees
SEQUENTIALLY. Each new tree studies the mistakes of the
previous one and tries to correct them. This process is called
boosting and is why XGBoost almost always outperforms Random
Forest.

It is the most widely used algorithm in machine learning
competitions and real world industry applications.

Why we switched:
- More accurate than Random Forest on most datasets
- Handles large datasets efficiently
- Better at capturing complex non-linear patterns
- Only required changing 2 lines of code

Settings used:
- n_estimators = 200 (200 sequential trees instead of 100)
- learning_rate = 0.1 (how aggressively each tree corrects errors)
- max_depth = 6 (how deep each tree can grow)
- eval_metric = logloss (for classification — penalizes confident wrong predictions)
- random_state = 42
- n_jobs = -1

ITERATION 3 — XGBOOST + FEATURE ENGINEERING (Final Version)
----------------------------------------------------------------

What is Feature Engineering?
Feature engineering means creating NEW columns from existing ones
to give the model extra signal — combined patterns it might miss
when looking at features individually.

The 4 new features we created:

1. academic_strength = gre_total * undergrad_gpa
   A student with both a high GRE AND high GPA is much stronger
   than someone who excels at only one. Multiplying them captures
   this combined academic capability in a single number.
   e.g. GRE=320, GPA=3.8 → academic_strength = 1216 (very strong)
   e.g. GRE=290, GPA=3.0 → academic_strength = 870 (average)

2. application_strength = sop_strength + (lor_avg_strength * 2) + lor_count
   Combines Statement of Purpose quality, Letter of Recommendation
   quality and quantity into one application package score.
   LOR average strength is weighted double because quality matters
   more than just having many letters.

3. research_power = research_experience_years + (publications_count * 3) + (conference_papers * 2)
   Especially important for PhD admissions. Publications are
   weighted 3x because a published paper is a much harder
   achievement than just having research experience.
   Conference papers weighted 2x for active field participation.

4. overall_profile_score = (academic_strength / 1000) + application_strength + research_power
   A single holistic number summarizing the entire applicant
   profile. Dividing academic_strength by 1000 normalizes it to
   a comparable scale with the other components.

Why feature engineering helps:
The model sees individual columns like GRE and GPA separately.
Feature engineering tells the model explicitly that the
COMBINATION of these things matters, not just each one alone.
This is especially true for PhD admissions where research
experience combined with publications is far more predictive
than either alone.

MASTERS Results:
  Regression:
    MAE  : 0.0326  (same as XGBoost alone)
    RMSE : 0.0407  (same as XGBoost alone)
    R²   : 0.9220  (marginal improvement)
  Classification:
    Accuracy  : 95.38%

Verdict:
Feature engineering helped PhD more than Masters which makes
perfect sense — research power and academic strength are far
more important signals for PhD admissions than Masters.
These are our final production models.

WHY WE DIDN'T DO HYPERPARAMETER TUNING
----------------------------------------------------------------

What is Hyperparameter Tuning?
Hyperparameters are the settings that control how a model learns
(n_estimators, learning_rate, max_depth etc.). We picked these
manually based on general best practices. Hyperparameter tuning
uses an automated process called Grid Search to try every
possible combination of settings and find the absolute best ones
for our specific dataset.

How Grid Search works:
You define a grid of options like this:
  n_estimators  : [100, 200, 300, 500]
  learning_rate : [0.01, 0.05, 0.1, 0.2]
  max_depth     : [4, 5, 6, 7, 8]

It then tries every single combination:
  4 x 4 x 5 = 80 different combinations

And it uses Cross Validation — meaning each combination is
tested 5 times on different splits of data for reliability:
  80 combinations x 5 folds = 400 full model trainings

Why it takes 10 hours:
Each full model training on 185,000 records takes roughly
1-2 minutes. So:
  400 runs x 1.5 minutes = 600 minutes = roughly 10 hours

Why we skipped it:
1. We are already at 92.4% R² and 95.4% accuracy
2. Tuning might squeeze out maybe 0.5% to 1% more at best
3. It would take approximately 10 hours to run
4. Project deadline did not allow for this time cost
5. The gains simply were not worth the time investment

The current results are production quality and more than
sufficient for the project requirements.
